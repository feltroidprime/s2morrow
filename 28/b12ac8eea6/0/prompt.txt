## PHASE 1: THE "ZERO BLIND SPOTS" STATE ARCHITECTURE

This is the **soul of the dApp UX**. Before any visual work, architect the state layer so that every possible user scenario is modeled, typed, and handled.

### 1A — Wallet Connection State Machine
Model wallet state as a **finite state machine** using Effect + atoms. Every state must have a corresponding UI treatment:

```
DISCONNECTED
  → User sees: Welcome state. Clear, warm CTA to connect. Explain what the dApp does in one sentence.
  → UI: Primary glass card, centered. No chrome. Just the value proposition + connect button.

CONNECTING
  → User sees: Subtle pulse animation on the connect button. "Connecting to wallet…" micro-copy.
  → UI: Button enters loading state. Rest of UI stays calm (no full-page loader).

CONNECTED (wrong network)
  → User sees: Gentle banner — "You're on [network]. Switch to Starknet Mainnet/Sepolia to continue."
  → UI: The banner is informational, not alarming. One-tap network switch button.

CONNECTED (correct network)
  → User sees: Full app unlocked. Wallet address displayed as truncated + ENS/StarkName if available.
  → UI: Smooth reveal animation of the main interface. The transition should feel like a curtain lifting.

CONNECTION_ERROR
  → User sees: Friendly message — "Couldn't connect. This usually means [reason]. Try again?"
  → UI: The error is contextual (inline near the connect button), not a toast that disappears.

DISCONNECTING
  → User sees: Brief fade-out of authenticated UI. Graceful return to welcome state.

SESSION_EXPIRED / ACCOUNT_CHANGED
  → User sees: Soft notification — "Your wallet changed. Refreshing your session…" Auto-reconnect.
```

### 1B — Transaction Lifecycle State Machine
Every on-chain write must go through this **full lifecycle**, and the UI must reflect each phase distinctly:

```
IDLE
  → Button is ready. No residual state from previous transactions.

VALIDATING
  → Frontend validates inputs BEFORE sending to wallet.
  → If validation fails: inline field-level errors with helpful copy (not "invalid input" — say WHY).
  → Effect pipeline: Parse → Validate → Simulate (estimate_fee) → Present confirmation.

SIMULATING
  → "Estimating gas…" shown subtly. If simulation fails (contract would revert):
  → Show the REASON in human language: map contract error codes to friendly messages.
  → Example: `Error::InsufficientBalance` → "You don't have enough [token] to complete this. You need [amount] more."

AWAITING_SIGNATURE
  → "Confirm in your wallet" — with a subtle animated indicator (pulsing wallet icon).
  → If the user takes too long (>30s), show gentle nudge: "Waiting for your confirmation in [Wallet Name]…"
  → If user rejects: "Transaction cancelled. No worries — nothing was sent." Return to IDLE cleanly.

PENDING (tx submitted, awaiting inclusion)
  → "Transaction submitted! Waiting for confirmation…"
  → Show a progress indicator. Link to block explorer (Starkscan/Voyager).
  → The user CAN navigate away. If they do, the pending state persists globally (atom) and shows as a non-intrusive banner/badge.
  → Poll or subscribe for tx receipt.

CONFIRMING (tx included, awaiting finality)
  → "Transaction confirmed on L2. Finalizing…"
  → Update UI optimistically where safe. Show "pending" badge on affected data until finality.

SUCCESS
  → Celebratory but tasteful. Brief success message with summary of what happened.
  → "You swapped 100 USDC → 0.05 ETH" — specific, not generic "Transaction successful."
  → Auto-dismiss after 5s, or user can dismiss. Affected data refreshes automatically.

FAILED (reverted on-chain)
  → Map the revert reason to a human message. Never show raw hex or "execution reverted."
  → Offer clear next steps: "This may have failed because [reason]. You can [retry / adjust amount / check balance]."
  → Log the raw error to console for debugging, but the user sees only the friendly version.

STUCK (pending for too long)
  → If no receipt after reasonable timeout: "This is taking longer than usual. Starknet may be congested."
  → Offer: "You can wait, or check the status on [explorer link]."
  → NEVER pretend the tx is lost. Be honest, be calm.
```

### 1C — Data Loading States
Every piece of on-chain data displayed must handle:

```
LOADING (first load)    → Skeleton shimmer (glass-style, not gray boxes). Matches final layout shape.
LOADED                  → Data rendered. Smooth fade-in from skeleton.
REFRESHING (background) → Stale data stays visible. Subtle refresh indicator (not a spinner replacing content).
ERROR                   → Inline error with retry: "Couldn't load [thing]. [Retry button]." Never a blank screen.
EMPTY                   → Friendly empty state with context: "No transactions yet. Your history will appear here after your first swap."
STALE (known outdated)  → Show data with subtle "updating…" indicator. Never show nothing while re-fetching.
```

### 1D — Effect Pipeline Architecture
All contract interactions MUST follow this pattern:

```typescript
// EXAMPLE — every contract write follows this shape:
const executeSwap = (params: SwapParams) =>
  Effect.gen(function* () {
    // 1. Parse & validate inputs (typed errors)
    const validated = yield* parseSwapParams(params)

    // 2. Simulate (catch contract reverts before wallet prompt)
    const simulation = yield* simulateTransaction(validated)

    // 3. Present fee estimate to user, await confirmation
    yield* awaitUserConfirmation(simulation.estimatedFee)

    // 4. Execute
    const txHash = yield* submitTransaction(validated)

    // 5. Wait for receipt
    const receipt = yield* waitForReceipt(txHash)

    // 6. Return typed result
    return receipt
  }).pipe(
    // Map every failure mode to a user-facing message
    Effect.catchTags({
      ParseError: (e) => Effect.fail(new UserFacingError({ message: `Invalid input: ${e.field} — ${e.reason}` })),
      SimulationFailed: (e) => Effect.fail(new UserFacingError({ message: mapRevertReason(e.reason) })),
      UserRejected: () => Effect.fail(new UserFacingError({ message: "Transaction cancelled." })),
      TxFailed: (e) => Effect.fail(new UserFacingError({ message: mapRevertReason(e.reason) })),
      NetworkError: () => Effect.fail(new UserFacingError({ message: "Network issue. Check your connection and try again." })),
      Timeout: () => Effect.fail(new UserFacingError({ message: "Taking longer than expected. Check the explorer for status." })),
    })
  )
```

**Rule:** If you find a contract call anywhere in the codebase that does NOT follow this pattern (raw `account.execute()` in a component, untyped errors, missing simulation step), refactor it immediately.

## PHASE 2: THE "LUCID" DESIGN FRAMEWORK (STRICT VISUAL RULES)

Every line of CSS or component code must strictly adhere to these principles:

### 2A — Materials over Colors
- Rely on translucency. Use backdrop filters (`backdrop-blur-xl`, `backdrop-blur-2xl`) combined with ultra-low opacity backgrounds (`bg-white/40`, `bg-black/10`) to create glass layers.
- Chain-specific accent colors (Starknet blue/purple) used sparingly — only for primary CTAs and active states. Never as backgrounds.

### 2B — Typography
- `Inter` or equivalent premium sans-serif via `next/font`.
- Strict optical tracking: tighter for headers (`tracking-tight`), standard for body.
- Hierarchy through **opacity** (`text-foreground/60` for secondary) more than size changes.
- Monospace for addresses, hashes, amounts: `font-mono tabular-nums` for perfect alignment.
- Numbers that change (balances, prices) use `tabular-nums` to prevent layout shift.

### 2C — Borders & Radii
- Generous radii (`rounded-2xl`, `rounded-3xl`).
- Borders practically invisible: `border border-white/15` to catch light on glass cards.
- No hard dividers between sections — use whitespace and subtle opacity changes.

### 2D — Shadows
- Deep, soft, multi-layered. Never harsh `shadow-md` defaults.
- Custom: `shadow-[0_8px_32px_rgba(0,0,0,0.08)]` style — diffuse, atmospheric.
- Elevated elements (modals, dropdowns) get progressively deeper shadows.

### 2E — Whitespace as Architecture
- Double the current padding/margins everywhere. The interface must breathe.
- Minimum `p-6` on card interiors, `gap-6` between sections.
- The most important elements get the most space around them.

### 2F — Motion & Transitions
- Spring-based, fluid animations. Nothing snaps; everything glides.
- Custom easing: `cubic-bezier(0.32, 0.72, 0, 1)` for entrances.
- Transaction state changes get smooth crossfade transitions, not instant swaps.
- Skeleton → Content transitions use staggered fade-ins (each element 50ms after the previous).
- **Wallet connection reveal:** Main interface slides/fades in with a choreographed sequence.
- **Number changes:** Balances and amounts animate between values (count-up/morph), never jump.

### 2G — dApp-Specific Visual Rules
- **Addresses:** Always truncated (`0x1234…abcd`), always copyable (click-to-copy with micro-feedback).
- **Token amounts:** Right-aligned, monospaced, with token icon left of the amount. Consistent decimal places.
- **Gas estimates:** Shown in both token amount AND approximate fiat value. Never just raw gas units.
- **Explorer links:** Subtle, secondary — available but not screaming. Icon + "View on explorer" on hover.
- **Pending transactions:** Persistent, non-intrusive. A small fixed indicator (bottom-right or top bar) that expands on hover to show pending tx details.

## CRITICAL DIRECTIVES

1. **No blind spots.** If a state is possible, it must be handled and beautiful. A user should never see a blank area, a spinner with no context, a cryptic error, or a UI that doesn't respond to their action.

2. **The dApp must feel robust.** Robustness is communicated through: consistent behavior, helpful error messages, smooth transitions between states, and the absence of flicker/jank/layout shift. Every interaction should feel like it was anticipated.

3. **The dApp must feel welcoming.** The welcome/connect state is the front door. It should be warm, clear, and explain the value in one glance. Wallet connection should feel like unlocking a door, not debugging a protocol.

4. **Never lost.** At any point, the user should know: where they are, what just happened, what's happening now, and what they can do next. Breadcrumbs, contextual headings, and clear state indicators are mandatory.

5. **Contract and frontend are one product.** If the contract doesn't give the frontend enough information to provide great UX (missing view functions, vague errors, missing events), that is a **bug to be fixed in the contract**, not a limitation to accept. Document all contract changes needed.

6. **Effect is the error backbone.** Every async operation, every contract call, every data fetch is an Effect pipeline with typed errors. No exceptions. No raw try/catch. No untyped Promise chains. This is non-negotiable — it's what makes the "zero blind spots" guarantee possible.

7. **Test what matters.** Every Effect pipeline gets a Bun test. Every state machine gets a Bun test. If a user can trigger a state, there's a test proving the app handles it correctly.
