# Context for VP-002: Effect Service Types, Errors, and WasmRuntime

## Ticket Scope
Create or align these files for the verify-playground category:
- `apps/demo/src/services/types.ts`
- `apps/demo/src/services/errors.ts`
- `apps/demo/src/services/WasmRuntime.ts`

Required outcomes:
- Branded `TxHash` and `ContractAddress`
- Domain types: `FalconKeypair`, `FalconSignatureResult`, `PackedPublicKey`, `VerificationStep`, `PipelineStep`
- Error classes: `WasmLoadError`, `KeygenError`, `SigningError`, `VerificationError`, `HintGenerationError`, `PackingError`, `StarknetRpcError`, `AccountDeployError`, `InsufficientFundsError`
- `WasmRuntime` implemented with `Effect.Service` (not `Context.Tag`), lazy-loads WASM from `/wasm/`, and avoids mutable module-level cache state

## Required Specs and Plans Reviewed

### `docs/specs/falcon-demo-website.md`
- P0 service requirements explicitly require:
  - Lazy WASM loading from `/wasm/`
  - Typed Effect service methods
  - `Schema.TaggedError` for all service failures
- Architecture constraints:
  - Crypto is browser-only (no server-side crypto)
  - Effect patterns mandatory (`Effect.Service`, `Schema.TaggedError`, `Effect.fn`, atoms)
  - Interactive sections are client-only and lazy-loaded (`ssr: false`)

### `docs/plans/2026-02-23-falcon-demo-website-design.md`
- Approved architecture requires a dedicated `WasmRuntime` dependency under the Effect services layer.
- Falcon service depends on runtime loader and exposes Effect-wrapped methods.
- WASM is loaded from `public/wasm/` and consumed in browser runtime.

### `docs/plans/2026-02-23-falcon-demo-website-impl.md`
- Task 3 originally scaffolds the exact `types.ts` and `errors.ts` payloads this ticket asks for.
- Planned `WasmRuntime.ts` uses `Context.Tag` + mutable `let cachedModule`, which this VP-002 ticket explicitly rejects.
- The plan code also uses unsafe casting (`as unknown as WasmModule`) which has already been corrected in current source via runtime validation.

### `CLAUDE.md`
Project conventions relevant here:
- Effect patterns are required and tests should use Bun.
- Keep implementations strongly typed and avoid ad hoc patterns that bypass Effect error typing.
- No server crypto behavior; browser-only crypto remains invariant.

## Key Implementation Implications for VP-002
- `types.ts` and `errors.ts` are already structurally aligned with the ticket and P0 requirements.
- Main delta is `WasmRuntime`: migrate from `Context.Tag` style to `Effect.Service` while preserving:
  - lazy dynamic import from `/wasm/falcon_rs.js`
  - typed `WasmLoadError` failures
  - runtime shape validation (`isValidWasmModule`)
  - no mutable module-level singleton cache
- Preserve compatibility for callers/tests currently doing Layer-based dependency injection with mock runtime modules.
- Keep `WasmModule` interface stable to avoid breaking `FalconService` and its tests.
