# Plan: VP-002 - Create Effect Service Types, Errors, and WasmRuntime

## Overview
This ticket finishes the verify-playground service foundation under Effect-TS conventions. The plan keeps `types.ts` and `errors.ts` aligned with the PRD while refactoring `WasmRuntime` from `Context.Tag` to `Effect.Service`, preserving lazy browser-only WASM loading from `/wasm/`, typed `Schema.TaggedError` failures, and zero mutable module-level cache state.

Implementation follows strict TDD: tests first, then minimal production changes per failing test.

## Risks and Mitigations
1. Dynamic import behavior differs across browser and Bun test runtime.
- Mitigation: test via injectable importer function instead of relying on real `import("/wasm/falcon_rs.js")`.

2. Refactor to `Effect.Service` may break existing layer injection patterns.
- Mitigation: keep service tag name `"WasmRuntime"`, preserve `WasmModule` interface, and keep `Layer.succeed(WasmRuntime, mock)` path in tests.

3. Cache semantics could regress to repeated imports.
- Mitigation: explicit importer call-count integration test in `WasmRuntime.test.ts` and forbid module-level mutable cache.

4. Duplicate/overlapping error tests may increase maintenance burden.
- Mitigation: keep focused scope in new tests; retain existing verify-playground and starknet error suites as integration coverage.
