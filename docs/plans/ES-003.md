# Plan: ES-003 - Implement WasmRuntime service with proper type validation

## Overview
This ticket implements the `WasmRuntime` Effect service using the `Context.Tag` pattern to load and provide the `falcon-rs` WASM module. It upgrades the previously planned implementation by removing unsafe type assertions (`as unknown as WasmModule`), replacing them with rigorous runtime validation of the expected WASM methods. Furthermore, it avoids mutable global state (`let cachedModule`) by relying on Effect's native Layer memoization to manage the WASM loading state properly. The service will fail cleanly with a typed `WasmLoadError` (a `Schema.TaggedError`).

## TDD Step Order
1. **Create Errors**: Define `WasmLoadError` in `apps/demo/src/services/errors.ts` (if not already existing).
2. **Write Tests FIRST**: Create `apps/demo/src/services/__tests__/WasmRuntime.test.ts` to test WasmRuntime using a mock module injected via `Layer.succeed`. Test both successful runtime validation and failure cases (missing methods).
3. **Implement Service**: Create `apps/demo/src/services/WasmRuntime.ts`. Implement the `WasmModule` interface, runtime validation function, and the `WasmRuntimeLive` Layer.

### 2. `apps/demo/src/services/WasmRuntime.ts` (Create)
- Define `WasmModule` interface:
  ```typescript
  export interface WasmModule {
    readonly keygen: (seed: Uint8Array) => { sk: Uint8Array; vk: Uint8Array }
    readonly sign: (skBytes: Uint8Array, message: Uint8Array, salt: Uint8Array) => { signature: Uint8Array; salt: Uint8Array }
    readonly verify: (vkBytes: Uint8Array, message: Uint8Array, signature: Uint8Array) => boolean
    readonly create_verification_hint: (s1: Int32Array, pkNtt: Int32Array) => Uint16Array
    readonly pack_public_key_wasm: (pkNtt: Uint16Array) => string[]
    readonly public_key_length: () => number
    readonly salt_length: () => number
  }
  ```
- Define `WasmRuntime` Tag:
  ```typescript
  import { Context, Effect, Layer } from "effect"
  export class WasmRuntime extends Context.Tag("WasmRuntime")<WasmRuntime, WasmModule>() {}
  ```
- Implement `isValidWasmModule(mod: unknown): mod is WasmModule`.
- Implement `WasmRuntimeLive` using Effect-native state (Layer memoization ensures it's a singleton when provided to the application). We'll use `Layer.effect` which memoizes by default.

### 3. `apps/demo/src/services/__tests__/WasmRuntime.test.ts` (Create)

- Test validation logic directly (valid and invalid modules).

## Tests to Write (Unit + Integration)
- **Unit Test: Validation logic**: Provide various objects to `isValidWasmModule` (null, missing `sign`, wrong type for `keygen`) and assert it returns false.
- **Integration Test: Layer Injection**: Use `Layer.succeed(WasmRuntime, mockModule)` and ensure a dependent effect can resolve the methods.
- **Integration Test: Live Layer Failure**: Test `WasmRuntimeLive` in an environment where the import fails or returns an invalid module, expecting a `WasmLoadError`.

## Risks and Mitigations
- **Risk**: Dynamic import `/wasm/falcon_rs.js` fails in the test environment (Node/Bun vs Browser).
  - **Mitigation**: Mock the dynamic import in tests or focus tests on the validation logic and Layer injection with mocks.
- **Risk**: WASM module exports methods with slightly different names.
  - **Mitigation**: The runtime validator will catch this immediately at runtime, throwing a clear `WasmLoadError`.
